# ZenithForge vLLM Reference Rearchitecture Plan

## Purpose

This document defines the rearchitecture plan for `ZenithForge` using `vLLM` as a serving reference, not as a clone target.

The target outcome is:

- keep `exllamav3` as the primary generation backend
- preserve inherited `tabbyAPI` value only where it survives review
- rebuild OpenAI-compatible serving around clearer contracts
- move from YAML-first startup to CLI-first startup
- make broader feature parity possible without turning the codebase into a `vLLM` copy

## Product Boundary

`ZenithForge` is:

- a separately branded product
- derived from `tabbyAPI`
- architecturally informed by `vLLM`
- built primarily around `exllamav3`

`ZenithForge` is not:

- an official `tabbyAPI` release
- an official `vLLM` distribution
- a renamed fork that keeps upstream semantics by accident

Practical rule:

- attribution to `tabbyAPI`, `vLLM`, and `exllamav3` stays explicit
- public product identity remains `ZenithForge`

## Planning Standard

This plan is written against the stricter standard requested for this project:

- do not scope features out just because `exllamav3` does not support them today
- define the full serving surface first
- separate `server-layer implementation`, `fallback backend`, `backend PR`, and `non-goal`
- do not hide unsupported capabilities behind silent omission

## What vLLM Gets Right

The key `vLLM` ideas worth carrying over are structural, not literal:

- app bootstrap is separate from protocol services
- render/preprocess is separate from backend execution
- parser registries are explicit
- streaming is assembled by stateful emitters, not ad-hoc SSE formatting
- endpoint registration is capability-aware
- serving surfaces are grouped by function family, not by utility module accidents

The most important architectural lesson is:

- routers should route
- services should implement behavior
- backends should expose capabilities
- streaming should be tested as a contract

## Current ZenithForge Problems

The current inherited stack is functional in places but structurally weak.

### 1. Router-heavy chat path

`endpoints/OAI/router.py` still orchestrates:

- inline model loading
- template application
- generation mode branching
- stream vs non-stream selection
- response transport

That is too much responsibility for a router.

### 2. Template ownership is in the backend

`backends/exllamav3/model.py` still participates directly in prompt-template ownership.
That couples serving semantics to backend load state.

### 3. Chat streaming is monolithic

`endpoints/OAI/utils/chat_completion.py` mixes:

- rendering
- stream assembly
- usage handling
- tool-calling fallback behavior
- disconnect handling
- parser resolution

This is too much logic for one utility path and makes regressions hard to isolate.

### 4. Reasoning parser wiring is incomplete

The parser registry exists, but the serving path is not cleanly wired end to end.
There are unresolved references left in the current streaming path, which indicates the architecture is ahead of the implementation.

### 5. Tool calling is too fallback-driven

The inherited stack still leans on second-pass generation logic in cases where parser-native streaming should be the primary path.

### 6. Startup is still YAML-first

`common/tabby_config.py` and the current docs still treat `config.yml` as the main control surface.
That is the opposite of the desired long-term operational contract.

### 7. Auth is file-centric

`api_tokens.yml` is still treated as a normal operating path.
That does not match the intended `vLLM`-style CLI-first runtime model.

## Rebuild Goals

The rearchitecture is complete only when all of the following are true.

- `/v1/chat/completions` uses the same render and parser pipeline for stream and non-stream
- reasoning separation works in both stream and non-stream paths
- tool-call parsing is deterministic across stream and non-stream paths
- `exllamav3` stays the primary text-generation backend
- backend template ownership is removed from the serving contract
- startup works from CLI alone without mandatory `config.yml`
- auth works from CLI/env/file inputs without mandatory `api_tokens.yml`
- broader endpoint families can be registered from capability state, not hard-coded assumptions
- rollout can be gated and rolled back with explicit feature flags

## Backend Strategy

### Primary rule

Do not make every serving feature pretend to be an `exllamav3` feature.

Instead, split backend responsibilities by capability family.

### Required backend families

- `TextGenerationBackendAdapter`
- `PoolingBackendAdapter`
- `SpeechBackendAdapter`
- `TokenizerAdapter`
- `RuntimeControlAdapter`

### Primary and fallback policy

For text generation:

- primary backend: `exllamav3`
- no `exllamav2` fallback in the new product plan
- if no working text backend exists, startup must fail clearly

For embeddings and ranking:

- primary backend: explicitly configured pooling or ranking backend
- current realistic fallback start point: `infinity` for embeddings only
- if capability is absent, the endpoint must fail clearly rather than pretend to work

For speech:

- primary backend: explicitly configured speech backend
- no fake speech layer over text-generation backends

### Unsupported capability behavior

When a requested capability is not implemented for the active server configuration, the behavior must be explicit.

That means one of the following:

- endpoint disabled by operator choice
- endpoint registered but returns `501 Not Implemented` with a precise reason
- endpoint enabled in degraded mode with documented limitations

Silent omission and silent no-op behavior are both prohibited.

## Target Architecture

### 1. Bootstrap layer

Bootstrap should build only the runtime graph:

- `ServeConfig`
- `RuntimeState`
- backend capability registry
- auth provider
- template registry
- parser registries
- endpoint policy registry

The bootstrap layer should no longer rely on module-global config as the source of truth.

### 2. Runtime state

`RuntimeState` should own:

- loaded text-generation backend
- loaded embedding or pooling backend
- loaded speech backend if any
- model registry and active model metadata
- request registry
- parser registries
- template registry
- auth configuration
- feature flags

### 3. Thin routers

Routers should only do:

- HTTP or WebSocket transport binding
- request parsing
- dependency checks
- service dispatch
- transport response return

### 4. Service layer

Required service families:

- `ModelsService`
- `ChatService`
- `CompletionsService`
- `ResponsesService`
- `MessagesService`
- `EmbeddingService`
- `PoolingService`
- `SpeechService`
- `TokenizerService`
- `RuntimeControlService`

### 5. Render layer

The render layer must own:

- message normalization
- template resolution
- tool schema injection
- multimodal placeholder assembly
- request-level template overrides
- BOS/EOS handling

Suggested output type:

- `RenderedPrompt`
  - `prompt_text`
  - `token_ids` optional
  - `mm_embeddings`
  - `conversation`
  - `template_metadata`
  - `stop_strings`

### 6. Parser layer

Required parser components:

- `ReasoningParserRegistry`
- `ToolParserRegistry`
- `StreamParseSession`
- `ParserPolicy`

Stream parse order should be explicit:

1. raw backend delta
2. reasoning split
3. tool-call split and accumulation
4. public protocol delta emission

### 7. Streaming layer

Streaming must become a state machine rather than a string formatter.

Required pieces:

- `ChatStreamSession`
- `ResponsesStreamSession`
- `EmitterHelpers`
- `SSEEncoder`
- per-choice state tracking
- usage and finish tracking

### 8. Capability routing

Services should not inspect backend types directly.

They should ask for capabilities:

- `text_generation`
- `embedding`
- `pooling`
- `rerank`
- `speech_transcription`
- `speech_translation`
- `tokenization`
- `runtime_control`

## CLI-First Runtime Model

The long-term operator experience should look like:

```bash
zenithforge serve /path/to/model \
  --host 0.0.0.0 \
  --port 8000 \
  --backend exllamav3 \
  --chat-template templates/chatml.jinja \
  --reasoning-parser exaone4 \
  --tool-call-parser qwen3_coder \
  --enable-auto-tool-choice \
  --api-key sk-... \
  --admin-key adm-...
```

### Configuration precedence

- CLI is first-class
- env vars are secondary
- YAML is legacy compatibility or import/export only

### Auth inputs

- `--api-key`
- `--admin-key`
- `--api-key-file`
- `--admin-key-file`
- `ZENITHFORGE_API_KEY`
- `ZENITHFORGE_ADMIN_KEY`
- `--disable-auth`

### YAML policy

Global `config.yml` and `api_tokens.yml` should leave the default startup path.

`tabby_config.yml` should be reconsidered as one of the following:

- model-local manifest
- model-local preset
- legacy compatibility file

It should not remain the global server control plane.

## Endpoint Registration Policy

Each endpoint should have a declared policy state:

- enabled
- enabled but degraded
- registered and returns `501 Not Implemented`
- not registered by explicit operator choice

There should be no accidental endpoint disappearance caused by missing implicit globals.

## PR Plan

### PR 0. Brand and scope lock

- complete `ZenithForge` branding transition
- freeze product naming and attribution rules
- freeze serving capability inventory

### PR 1. CLI-first bootstrap

- introduce `serve` command structure
- define `ServeConfig`
- define config precedence
- separate startup config from module-global runtime state

### PR 2. RuntimeState and service boundaries

- introduce `RuntimeState`
- move router logic into service classes
- route core and compatibility endpoints through shared runtime commands

### PR 3. Render extraction

- extract chat rendering into an explicit service
- move template ownership out of backend containers
- make tokenization and render endpoints share the same render core

### PR 4. Text backend adapter

- define typed stream event contract
- wrap `exllamav3` behind `TextGenerationBackendAdapter`
- remove backend-specific dict contracts from serving code

### PR 5. Chat streaming rebuild

- replace monolithic chat stream path with `ChatStreamSession`
- isolate role chunks, tool deltas, reasoning deltas, finish chunks, usage chunks

### PR 6. Reasoning parser end-to-end wiring

- connect parser resolution to request and stream state
- make reasoning separation identical across stream and non-stream

### PR 7. Tool-call pipeline rebuild

- make parser-native stream parsing the default path
- isolate second-pass generation as explicit fallback policy where still needed

### PR 8. Legacy config and auth deprecation

- move `config.yml` and `api_tokens.yml` out of the default path
- document legacy mode clearly

### PR 9. Text-surface parity expansion

- add `/v1/responses`
- add `/v1/messages`
- add `/v1/messages/count_tokens`
- keep these on shared text-generation core

### PR 10. Render and tokenization parity

- add `/v1/chat/completions/render`
- add `/v1/completions/render`
- rebuild `/tokenize`, `/detokenize`, `/tokenizer_info`

### PR 11. Pooling and ranking backend family

- introduce `PoolingBackendAdapter`
- add embeddings, pooling, score, rerank, classify family support

### PR 12. Speech and realtime backend family

- introduce `SpeechBackendAdapter`
- add transcription and translation
- add realtime websocket session layer

### PR 13. Observability and ops parity

- add or rebuild `/metrics`, `/server_info`, `/load`, `/version`, `/ping`, `/invocations`
- add backend-hooked runtime ops where the capability exists

### PR 14. Optional token-level extensions

- decide on `/inference/v1/generate` and `/abort_requests` only after request registry and token stream core are mature

## Implementation Gates

The following must exist before any structural PR merges:

- `ServeConfig` draft
- `RuntimeState` draft
- backend adapter contracts
- render output contract
- streaming session state model
- capability registry draft
- rollback flag list
- golden transcript format

The following must exist before parity PRs merge:

- module-level `vLLM` contract review notes for the targeted feature family
- endpoint-specific golden tests
- at least one real backend smoke test for the family being merged

## Final Recommendation

The correct order is:

1. finish the `ZenithForge` identity transition
2. lock the capability inventory and backend-family strategy
3. rebuild bootstrap and runtime boundaries
4. rebuild render, parser, and streaming around `exllamav3`
5. expand parity family by family

The biggest current risk is not missing a single endpoint.
It is extending the inherited stack before the core serving contract is explicit.

So the first implementation work should still be structural.
But from this point forward, structural work must be tied to the full serving-surface plan, not just chat-only cleanup.
